# 问题与视觉编码

## 先用一句话说明

先说清要比较的关系，再决定用位置、长度、面积、颜色或形状表达数据。

图表类型不是从 API 名单中随便挑一个，而是对问题作出的编码选择。趋势、类别比较、分布、构成、相关关系和二维场回答的问题不同。先写一句读者应该从图中得出的结论，再选最直接的视觉通道，通常比先追求“看起来丰富”更可靠。

## 为什么重要

同一组数字可以画成许多图，但并非都同样容易读。位置最适合精确比较，长度适合从共同基线比较，面积和角度更适合少量的整体构成，颜色适合给位置再增加一个连续变量。把过多变量塞进颜色、大小、形状和透明度，会让读者不断查图例。Plot 提供多种系列和面板，是为了匹配不同任务，不是要求每张图都用上。

## 工作模型

先判断“观测单位”是什么，再判断要比较哪种关系。连续时间上的值通常用 LineSeries；离散但有保持语义的状态可用 StepSeries；独立样本用 ScatterSeries；分类大小用 BarSeries；一批样本的形状用 HistogramSeries、BoxPlotSeries 或 ViolinSeries；位置网格上的值用 HeatmapSeries 或 ContourSeries；少量整体构成才考虑 PieAxes；同量纲多指标轮廓才考虑 RadarAxes。

以下对比展示同样的季度数值如何因问题不同而编码。趋势关注顺序和变化，分类比较关注共同基线上的长度。

```cangjie role=contrast
let trend = LineSeries([1.0, 2.0, 3.0, 4.0], [12.0, 19.0, 15.0, 23.0], label: "收入")
axes.add(trend)
```

```cangjie role=trace
axes.setXScale(CategoryScale(["Q1", "Q2", "Q3", "Q4"]))
axes.showGridX = false
let bars = BarSeries([12.0, 19.0, 15.0, 23.0], label: "收入")
bars.showValues = true
axes.add(bars)
```

## 选择与取舍

折线会暗示相邻点之间存在连续或有意义的顺序，所以无序类别不要连线。柱形从共同基线比较长度很清楚，但类别过多时会拥挤，长名称可改为水平柱或筛选重点。散点最适合看两个连续变量的关系；如果只是时间序列，散点可能隐藏顺序。饼图要求所有切片构成一个整体，而且切片要少；精确比较仍优先柱形。雷达图强调轮廓，所有辐条应先归一化到可比较的范围。

表达不确定性时，误差棒适合每个位置的区间，置信带适合连续曲线，完整分布则用直方图、箱线图或小提琴图。二维场的颜色一定要配色标，发散色图只有在零点或基准值确有含义时使用。多面板往往比一张“全能图”更易懂，因为每个面板能保持明确的问题。

## 应用这个模型

在写代码前做一个四列小表：读者问题、观测单位、主要比较、所选编码。例如“哪个渠道转化高”对应渠道类别、转化率、共同基线长度，因此用柱形；“延迟是否随负载增加”对应服务样本、两个连续变量、位置关系，因此用散点；“预测随时间的不确定性如何变化”对应时间、上下界、位置加带宽，因此用折线加区间带。

对象安放规则见[Figure、面板与系列](figure-axes-series.md)。做出选择后，可按[选择图表](../how-to/choose-chart.md)的任务表落到具体 API。选择不必永久不变：先用最简单的编码回答核心问题，读者确实需要更多维度时再增加颜色或分面。

## 常见误解

“数据维度多就用雷达图”会混淆不同单位；“占比都用饼图”会让许多相近切片难以比较；“平滑曲线更真实”也不成立，平滑会补出采样点之间并不存在的形状。颜色不是装饰：连续数值需要连续色图，类别需要彼此可辨的离散颜色。图例也不能挽救不合适的编码；如果读者必须反复对照大量颜色，优先拆面板或直接标注。
