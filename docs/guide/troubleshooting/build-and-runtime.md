# 构建与窗口运行排错

## 先看现象

本页处理三类边界问题：编译器找不到 Plot 包，程序启动时找不到 SDL 或无法建立渲染环境，以及窗口打开后命令没有立即返回。先按错误发生阶段分类，不要同时改依赖、源代码和窗口逻辑。每个分支都给出可执行的检查动作与可观察的修复确认。

## 可能原因

依赖问题通常来自 `cjpm.toml` 的本地路径或包配置；原生运行问题通常来自 SDL 动态库、当前会话没有显示环境或运行目录不同；生命周期问题则常是把 `PlotWindow.show()` 的事件循环误认为死循环。`FigureExport` 虽然不显示窗口，也会创建隐藏渲染环境，因此原生依赖仍需可用。

## 诊断步骤

### 症状一：编译器提示找不到 `plot` 包或导入失败

先在示例项目目录运行 `cjpm build`，检查错误是否发生在解析依赖阶段。核对 `cjpm.toml` 中本地依赖路径是否指向包含 Plot `cjpm.toml` 的目录，并检查包名拼写。用下面探针把问题缩到最小：它只导入顶层版本常量，不涉及窗口或图片。若连它也不能编译，问题在依赖配置而不在图表 API。确认标准是构建退出码为 0，并打印版本号。

```cangjie role=probe
package guide_examples

import plot.VERSION

main(): Unit {
    println("plot version=${VERSION}")
}
```

依赖确认后，再用一个真正创建 Figure 的最小修复程序替换探针。它检查顶层类型、Series 子包和对象连接都能解析；不要把 GUI 加回来，直到这一步稳定通过。

```cangjie role=fix
package guide_examples

import plot.Figure
import plot.series.LineSeries

main(): Unit {
    let (figure, axes) = Figure.single(title: "依赖检查")
    axes.add(LineSeries([1.0, 2.0], [2.0, 3.0], label: "正常"))
    println("panels=${figure.panelCount()}")
}
```

### 症状二：编译成功，但启动时报 SDL、动态库或显示环境错误

先记录完整错误，不要只截最后一行。运行 `cjpm run` 时检查当前终端是否处于普通桌面会话，SDL 动态库目录是否在系统加载路径中，并与项目 README 要求的版本一致。再比较窗口示例和导出示例：两者都失败，优先检查原生库；只有窗口失败，则检查显示服务和远程会话能力。修复后确认窗口能实际显示，或导出程序生成可打开 PNG，并且进程退出码为 0。远程服务器没有桌面时，不要强行启动 PlotWindow，改用已配置好的无窗口渲染节点执行 FigureExport。

### 症状三：窗口已经显示，但终端一直不返回

检查代码是否在 `window.show()` 之后安排了打印、导出或测试断言。`show` 会运行事件循环，窗口关闭后才返回；只要窗口还能响应移动、缩放和关闭，这不是挂死。点击关闭按钮，确认进程随即退出。若窗口完全无响应，检查是否在事件回调或 `show` 前执行了长时间计算，并把数据准备移到构图前或后台任务。修复后应能看到窗口持续刷新、关闭操作生效、终端恢复提示符；需要自动完成的任务应直接调用 FigureExport，而不是等待人工关闭。

## 修复方法

依赖错误只修改项目清单和真实路径；不要把库源文件复制进应用。原生错误按操作系统配置动态库与显示环境；不要在代码里硬编码机器私有路径。生命周期错误通过拆分 `buildFigure(): Figure`、窗口入口和导出入口解决。先让纯构建器可调用，再分别验证窗口和文件交付，能显著缩小问题范围。

## 确认已经修复

按顺序确认：`cjpm build` 退出码为 0；最小探针打印版本或面板数；窗口示例能显示、响应并在关闭后返回；导出示例能生成非空、可打开的 PNG。只通过其中一步不代表全部环境正确。把实际命令、退出码和输出文件路径记录到项目验证日志，便于另一台机器复现。

## 避免再次发生

在仓库中保留一个最小编译探针和一个无窗口导出冒烟测试；升级工具链、SDL 或 Plot 后先运行它们。构图函数不打开窗口，命令行入口不依赖人工交互，GUI 入口明确说明 `show` 生命周期。CI 环境若不提供渲染能力，应显式跳过 GUI 运行而仍执行编译，不要把未运行误报成通过。
